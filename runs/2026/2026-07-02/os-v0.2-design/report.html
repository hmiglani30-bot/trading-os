Trading OS v0.2 Design Report

Date: 2026-07-02
Workflow: os-v0.2-design
Scope: Define how prompts, source selection, outputs, and browser-readable reports work together.

1. Executive Summary

The first scaffold was useful as a file structure, but it was not strict enough about prompt quality, source selection, and output storage. Version 0.2 upgrades the system around three principles:

  1. Prompts must be source-aware.
  2. Workflows must produce one readable report, not many scattered outputs.
  3. Every completed run must be saved as Markdown, rendered to HTML, and archived as PDF.

3. Operating Model

user input + source matrix
  -> prompt module
  -> workflow report
  -> runs/YYYY/YYYY-MM-DD/<workflow>/report.md
  -> generated report.html and report.pdf
  -> site/index.html browser index

4. Report Output Contract

Every workflow should create this folder shape:

runs/YYYY/YYYY-MM-DD/<workflow-name>/report.md
runs/YYYY/YYYY-MM-DD/<workflow-name>/report.html
runs/YYYY/YYYY-MM-DD/<workflow-name>/report.pdf

report.md is the source of truth. report.html is the browser-readable version. report.pdf is the archive/shareable version.

6. Source Policy

The source policy is deliberately tiered:

  1. Primary or official sources for precise facts.
  2. Dashboards for scanning and context.
  3. News and commentary for narrative, not as source-of-truth data.

Each report must include Sources Used and Missing Data sections.

7. Browser Output Path

After merge, enable GitHub Pages with source set to GitHub Actions. Then run the workflow named Render and publish reports. It will generate the HTML/PDF outputs and publish the browser index.

8. Current Limitations

  • The GitHub connector can create text files, but binary PDF files should be generated by GitHub Actions or locally.
  • Some finance-specific wording can trigger connector safety checks, so repo files use research/reporting language and avoid execution instructions.
  • This system stores research outputs; it does not automate financial account actions.
